<p>Since its launch in 2006, <a href="https://aws.amazon.com/s3/" rel="noopener noreferrer" target="_blank">Amazon S3</a> has used a global namespace where bucket names must be unique across all <a href="https://docs.aws.amazon.com/accounts/latest/reference/accounts-welcome.html" rel="noopener noreferrer" target="_blank">AWS accounts</a> and <a href="https://aws.amazon.com/about-aws/global-infrastructure/regions_az/" rel="noopener noreferrer" target="_blank">AWS Regions</a>. This design has served customers well at scale, but organizations managing multiple accounts and environments often encounter naming collisions. When a bucket is deleted, its name returns to the global pool and could be claimed by another account, which requires careful handling when applications, DNS records, or automation still reference it. To work around these constraints, teams typically add random suffixes, manually embed account identifiers, or adjust deployment workflows when a desired name is unavailable. At enterprise scale, a more structured approach to bucket naming simplifies operations and strengthens security posture.</p> 
<p>S3 introduced <a href="https://aws.amazon.com/blogs/aws/introducing-account-regional-namespaces-for-amazon-s3-general-purpose-buckets/" rel="noopener noreferrer" target="_blank">account regional namespaces for general purpose buckets</a> in March 2026, an opt-in feature that scopes bucket names to your individual AWS account and region. No more racing against a shared global namespace. Your desired bucket names are available within your account and region, and unlike the traditional global namespace, deleted account regional names can’t be claimed by another account and remain reserved for your use. This consistent naming format simplifies <a href="https://aws.amazon.com/what-is/iac/" rel="noopener noreferrer" target="_blank">infrastructure as code (IaC)</a> templates across environments.</p> 
<p>For most customers, the recommended path forward is to create new buckets in the account regional namespace. Existing global namespace buckets continue to work exactly as they do today and migrating them is not a requirement. In this post, we walk you through how to migrate existing buckets using <a href="https://aws.amazon.com/s3/features/replication/" rel="noopener noreferrer" target="_blank">S3 Replication</a> for organizations that have specific reasons to do so, and how to enforce account regional namespace adoption across your organization using <a href="https://aws.amazon.com/organizations/" rel="noopener noreferrer" target="_blank">AWS Organizations</a> <a href="https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps.html" rel="noopener noreferrer" target="_blank">service control policies (SCPs)</a>.</p> 
<h2>When does migrating make sense?</h2> 
<p>From a features perspective, the account regional namespace works identically to global namespace buckets.</p> 
<ul> 
 <li><strong>Consolidate to a single namespace model</strong> – If you want every bucket in your organization to follow the same naming convention for consistency in automation, <a href="https://aws.amazon.com/iam/" rel="noopener noreferrer" target="_blank">AWS Identity and Access Management (IAM)</a> policies, and documentation, migrating existing buckets avoids the split between two naming schemes.</li> 
 <li><strong>Governance you can automate </strong>– With the <code>s3:x-amz-bucket-namespace</code> condition key, your security team can use SCPs to block global namespace bucket creation across the organization. A naming policy becomes an infrastructure guardrail. We cover the phased enforcement approach later in this post.</li> 
 <li><strong>Decommission legacy buckets safely </strong>– If you need to retire workloads that use global namespaces, migrating the data to a new bucket with an account regional namespace first lets you delete the global bucket. Note that deleted global bucket names can potentially be claimed by other accounts, so only delete them after all references (DNS records, application configs, IAM policies) have been updated.</li> 
 <li><strong>Ownership visibility in AWS CloudTrail logs </strong>– When every bucket name embeds the account ID and Region, your security team can identify bucket ownership from the name alone in <a href="https://aws.amazon.com/cloudtrail/" rel="noopener noreferrer" target="_blank">AWS CloudTrail</a> logs, without cross-referencing account inventories.</li> 
</ul> 
<p>For many organizations, the right starting point is enforcing account regional namespaces for new buckets using SCPs and only migrating existing buckets where there is a clear operational reason to do so.</p> 
<h2>Planning your migration</h2> 
<p>Because existing buckets can’t be renamed, migration requires creating a new bucket and transitioning workloads to it. A little upfront planning makes this straightforward.</p> 
<h3>Inventory buckets</h3> 
<p>Use <a href="https://aws.amazon.com/config/" rel="noopener noreferrer" target="_blank">AWS Config</a> or <a href="https://aws.amazon.com/s3/storage-lens/" rel="noopener noreferrer" target="_blank">S3 Storage Lens</a> to generate a list of all existing buckets across accounts in your organization. For each bucket, capture the following information:</p> 
<ul> 
 <li>The bucket name and owner account and region.</li> 
 <li>All resources and configurations that reference the bucket. Common examples include IAM policies, bucket policies, <a href="https://aws.amazon.com/lambda/" rel="noopener noreferrer" target="_blank">AWS Lambda</a> functions, <a href="https://aws.amazon.com/sagemaker/" rel="noopener noreferrer" target="_blank">Amazon SageMaker</a> pipelines, <a href="https://aws.amazon.com/bedrock/knowledge-bases/" rel="noopener noreferrer" target="_blank">Amazon Bedrock Knowledge Bases</a>, <a href="https://aws.amazon.com/cloudformation/" rel="noopener noreferrer" target="_blank">AWS CloudFormation</a> stacks, and application configs, but this will vary by workload.</li> 
 <li>Approximate data volume and access patterns. Use <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/cloudwatch-monitoring.html" rel="noopener noreferrer" target="_blank">Amazon CloudWatch S3 request metrics</a> to understand how frequently each bucket is accessed.</li> 
</ul> 
<h3>Define naming convention</h3> 
<p>Because the suffix is fixed (<code>accountid-region-an</code>), define a consistent prefix standard before migrating. You can use the following recommended pattern: <code>&lt;appname&gt;-&lt;purpose&gt;-&lt;accountid&gt;-&lt;region&gt;-an</code></p> 
<p>For example: <code>payments-logs-123456789012-us-east-1-an</code></p> 
<p>Keep in mind that the bucket name prefix and account regional suffix combined must be between 3 and 63 characters long. Plan your prefix length accordingly, because longer prefixes may exceed the limit once the suffix is appended.</p> 
<h3>Prioritize by risk</h3> 
<p>Start with sandbox and dev accounts. This provides the lowest area of impact and fastest feedback. Move to staging after dev migration is validated. Address production last, with a full cutover runbook tested in lower environments.</p>

<h2>Prerequisites</h2> 
<p>Before you begin, verify you have the following in place:</p> 
<ul> 
 <li>An <a href="https://signin.aws.amazon.com/signin?redirect_uri=https%3A%2F%2Fportal.aws.amazon.com%2Fbilling%2Fsignup%2Fresume&amp;client_id=signup" rel="noopener noreferrer" target="_blank">AWS account</a> with an existing S3 global namespace bucket you want to migrate</li> 
 <li><a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/Versioning.html" rel="noopener noreferrer" target="_blank">S3 Versioning</a> enabled on the source bucket (required for replication)</li> 
 <li>Permissions for <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/setting-repl-config-perm-overview.html" rel="noopener noreferrer" target="_blank">S3 Replication</a></li> 
 <li><a href="https://aws.amazon.com/organizations/" rel="noopener noreferrer" target="_blank">AWS Organizations</a> with SCPs enabled (if following the governance enforcement section)</li> 
</ul> 
<h2>Step 1: Create new account regional bucket</h2> 
<p>Create the new general purpose bucket using the <a href="http://aws.amazon.com/console" rel="noopener noreferrer" target="_blank">AWS Management Console</a>, <a href="https://aws.amazon.com/cli/" rel="noopener noreferrer" target="_blank">AWS Command Line Interface (AWS CLI)</a>, <a href="https://aws.amazon.com/what-is/sdk/" rel="noopener noreferrer" target="_blank">SDK</a>, or IaC tool of your choice. The AWS News Blog post, <a href="https://aws.amazon.com/blogs/aws/introducing-account-regional-namespaces-for-amazon-s3-general-purpose-buckets/" rel="noopener noreferrer" target="_blank">Introducing account regional namespaces for S3 general purpose buckets</a> walks through the S3 console experience, AWS CLI, <a href="https://docs.aws.amazon.com/boto3/latest/" rel="noopener noreferrer" target="_blank">Boto3</a>, and CloudFormation examples. For the full naming conventions, restrictions, and additional IaC patterns, refer to <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/gpbucketnamespaces.html" rel="noopener noreferrer" target="_blank">Namespaces for general purpose buckets</a>.</p> 
<p>When creating the new bucket, use the prefix you defined in your naming convention. The account regional suffix (<code>-{accountid}-{region}-an</code>) is appended automatically when you use <code>BucketNamePrefix</code>, or you can construct the full name explicitly.</p> 
<h2>Step 2: Set up S3 Replication</h2> 
<p>Configure S3 Replication from the old global bucket to the new account regional bucket. This keeps new data in sync continuously, giving you a safe window to update all consumers before cutting over. To copy historical data, create an <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-batch-replication-batch.html" rel="noopener noreferrer" target="_blank">S3 Batch Replication</a> job, which replicates existing objects that were in the bucket before the replication rule was enabled. Enable <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication-how-setup.html" rel="noopener noreferrer" target="_blank">live replication</a> first, then run <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-batch-replication-batch.html" rel="noopener noreferrer" target="_blank">batch replication</a> to confirm no objects are missed. Before configuring replication, confirm the destination bucket has versioning enabled. If the source bucket has <a href="https://aws.amazon.com/s3/features/object-lock/" rel="noopener noreferrer" target="_blank">S3 Object Lock</a> enabled, S3 Object Lock must also be enabled on the destination bucket. For setup instructions, see <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html" rel="noopener noreferrer" target="_blank">Replicating objects within and across Regions</a>.</p> 
<h2>Step 3: Update references</h2> 
<p>Update all resources and configurations that reference the old bucket to point to the new bucket name. This includes IaC templates, IAM and bucket policies, Lambda environment variables, pipeline and training job configs, knowledge base data source configurations, application configs, and environment secrets. The specific resources will vary by workload.</p> 
<h2>Step 4: Transition to the new bucket and retire old bucket</h2> 
<p>Turn on CloudWatch S3 request metrics on the old bucket and monitor until the request count reaches zero, confirming no active consumers remain. After traffic has stopped, keep the old bucket for a grace period (14 days is a reasonable default). After the grace period, disable replication rules after you confirm that the new bucket is the source of truth. The safest approach is to keep the old global namespace bucket indefinitely, even if empty, to prevent the name from being claimed by another account. If you choose to delete it, audit all references (application configs, DNS records, IAM policies, third-party integrations) thoroughly, because the name will return to the global pool and could be reused by anyone.</p> 
<h2>Step 5: Enforcing governance with SCPs</h2> 
<p>After your accounts are migrated, lock in the new standard. For ready-to-use policy examples for SCPs, <a href="https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_rcps.html" rel="noopener noreferrer" target="_blank">Resource Control Policies (RCPs)</a>, and IAM policies, see <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/gpbucketnamespaces.html#require-account-regional" rel="noopener noreferrer" target="_blank">Requiring the creation of buckets in your account regional namespace</a>. These policies use the <code>s3:x-amz-bucket-namespace</code> condition key to deny <code>CreateBucket</code> calls that don’t specify the account regional namespace.</p> 
<p>The question is not which policy to use, but when to apply it. Enforcing too early breaks teams that have not yet migrated. Enforcing too late allows new global namespace buckets to accumulate. Tie enforcement to your migration progress:</p> 
<ul> 
 <li><strong>Phase 1</strong> – Apply to sandbox and new account organizational units (OUs) immediately. There are no existing buckets to migrate.</li> 
 <li><strong>Phase 2</strong> – Apply to dev and staging OUs after migration is complete and validated.</li> 
 <li><strong>Phase 3</strong> – Apply organization-wide after production migration is confirmed.</li> 
</ul> 
<h2>Cost considerations</h2> 
<p>The only incremental spend during migration is standard S3 storage and replication request charges while both the old and new buckets exist in parallel. This is temporary. After the old bucket is retired, your storage bill returns to baseline. For most workloads, the migration window is measured in days.</p> 
<h2>Cleaning up</h2> 
<p>If you created test buckets during the walkthrough, complete the following steps to clean up your resources:</p> 
<ol> 
 <li>On the S3 console, select the test bucket you created.</li> 
 <li>Empty the bucket, then delete it.</li> 
 <li>If you deployed a test SCP, remove it from the OU on the Organizations console.</li> 
</ol> 
<h2>Conclusion</h2> 
<p>Account regional namespaces change how Amazon S3 bucket identity works. You don’t have to migrate your entire S3 data estate to benefit. You can enforce account regional namespaces for new buckets immediately using SCPs and migrate existing buckets on your own timeline. For buckets you do migrate, the process is a one-time investment. Create new buckets, replicate data, update references, transition over, and enforce with SCPs. After you complete this process, your bucket names are permanently yours, uniform across every environment, and ownership is visible directly in CloudTrail logs.</p> 
<p>Start with your sandbox accounts today and implement the SCP enforcement through your AWS Organization as each environment is migrated.</p>
